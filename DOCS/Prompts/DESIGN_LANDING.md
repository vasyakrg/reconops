# DESIGN_LANDING — дизайн-бриф для лендинга recon.ru

> Задание для дизайнера/Claude. Цель — single-page промо-сайт продукта,
> который понятно, дружелюбно и красочно объясняет потенциальным
> заказчикам, что такое Recon и зачем он им. Перед работой обязательно
> читать `README.md`, `DOCS/Prompts/PROJECT.md` §1–§3, `DESIGN_HUB.md`
> (для визуальной консистентности с продуктом).

---

## 0. TL;DR для дизайнера

Лендинг продаёт **доверие к незнакомому классу инструмента**: «ИИ ходит
по вашему проду». Главное возражение читателя — «он что-то сломает».
Главный аргумент — «не сломает, потому что физически не может». Лендинг
строится вокруг этого диалога.

**Визуально:** тёмная база (консистентно с хабом), но с яркими
SVG-иллюстрациями, цветными accent-блоками, анимированными
диаграммами. Не «мрачный enterprise», а «живой dev-tool вечерним
светом» — Tailscale, Fly.io, Linear marketing.

**Тон:** friendly + доступный. Не sales-bullshit, но и не суровый
engineering manual. Можно с историей («три часа ночи, пейджер сработал»),
можно с метафорами («второй пилот, которого вы можете остановить
кнопкой»). Жаргон допустим в технических секциях, в hero — на
человеческом языке.

**Продажи:** одной строкой в конце, без CTA-баннеров. *«Заинтересовало?
Напишите нам: hello@recon.ru»*. Никакого «Book a demo», «Get a quote»,
«Contact sales».

## 1. Зачем существует лендинг

- Объяснить за **30 секунд** (hero+первый scroll), что Recon делает и
  почему он не опасен.
- Дать **технически грамотному читателю** достаточно деталей, чтобы он
  мог уйти и сам решить «хочу попробовать» (без обязательного контакта
  с sales).
- Снять страх перед «ИИ на проде» — через демонстрацию read-only
  гарантий и step-by-step approve flow.
- Привлечь правильную аудиторию (см. §2) и отфильтровать неправильную
  (мы не для тех, кто хочет «AIOps automated remediation» — мы про
  controlled investigation).

## 2. Аудитория (ICP)

В порядке приоритета:

1. **SRE/DevOps лиды** в командах с парком 10–500 Linux-хостов
   (k8s nodes, edge servers, bare metal). Технари, читают сами,
   решение принимают вместе с CTO. **Главные читатели лендинга.**
2. **CTO / Head of Infrastructure** в средних компаниях (50–500
   инженеров). Решают бюджет, оценивают риск. Им важно: безопасность,
   self-hosted, no data exfiltration, на каком LLM-провайдере.
3. **Security инженеры** — будут проверять read-only гарантии и
   аудит-трейл. Им важно: 5 слоёв, mTLS, audit log, retention.
4. **Случайный зашедший разработчик** — может уйти быстро, но если
   останется, может пошарить ссылку SRE-другу. Под него рассчитан
   tone-of-voice hero-секции.

**Не наша аудитория:** малый бизнес без выделенной инфры, AIOps-energy
людей которые ждут полностью автоматических action'ов, разработчики
SaaS без on-prem задач.

## 3. Tone of voice

### 3.1. Принципы

- **Антитеза маркетинговому bullshit.** Не «Revolutionary AI-powered
  observability», а «ИИ предлагает следующий шаг — вы решаете,
  выполнять ли его».
- **Конкретные числа и термины** в технических секциях. «11 встроенных
  коллекторов», «mTLS между агентом и хабом», «контекст ~150K токенов
  с компактификацией», «один Go binary 28MB».
- **Истории и метафоры** в hero и use cases. «Это как code review для
  каждой команды на проде».
- **Никакого FOMO**. Никаких «Join 1000+ teams» (мы только запускаемся
  и врать неприлично). Никаких countdown-таймеров, sticky CTA-bars,
  exit-intent попапов.
- **Самоуверенность без хвастовства.** «Read-only enforced на 5 слоях»
  — факт. Не «World's safest» — это пустота.

### 3.2. Примеры фраз

**Hero (✅ так):**

> ИИ-расследования инцидентов на Linux-парке. Каждый шаг — под вашим
> контролем. Read-only по конструкции — не сможет ничего сломать.

**Hero (❌ не так):**

> Революционная AI-платформа для observability нового поколения,
> которая трансформирует ваш incident management workflow.

**О безопасности (✅):**

> Пять слоёв read-only enforcement. Если хотя бы один пропустит
> destructive команду — она упрётся в следующий. Подробности — ниже.

**О безопасности (❌):**

> Enterprise-grade security with industry-leading compliance.

### 3.3. Язык

- Русский основной. UI продукта тоже русский — консистентно.
- Англицизмы в терминах допустимы и желательны: SRE, DevOps, k8s,
  DNS, mTLS, LLM, observability, audit log, post-mortem. Не пытаться
  переводить «оператор-исследователь машинного обучения».
- Имена продуктов — на латинице: Recon, не «Рекон». OpenRouter, kubectl,
  Grafana, Datadog.

## 4. Политика по продажам

Жёстко. **На лендинге нет ни одного `[Book a demo]` баннера.**

Единственное упоминание контакта:
- В footer одной строкой: «Заинтересовало? Напишите нам:
  [hello@recon.ru](mailto:hello@recon.ru)»
- Опционально: тот же текст в самом конце последней content-секции
  (после FAQ), без отдельного блока.

Что **есть** для активации читателя:
- Ссылка на GitHub репозиторий (если open-source) или на «request
  early access» mailto.
- Ссылка на quickstart раздел `README.md` («попробовать за 5 минут»).
- Ссылка на полную архитектурную документацию (`PROJECT.md`).

Что **категорически нет**:
- Pricing страницы (на этом этапе).
- Форм с email subscription.
- «Compare plans» таблиц.
- Customer logos карусели (нет клиентов и не врём).
- Testimonials.
- «Trusted by» секций.

## 5. Информационная архитектура

Single-page scroll. Sticky-nav сверху с якорями.
Секции в порядке скролла:

### 5.1. Hero — **P0**

**Слева:** название Recon, одностроковый промис, подзаголовок (2
предложения), две кнопки: `Документация` (→ якорь #docs или GitHub),
`Quickstart 5 минут` (→ якорь #quickstart). На уровне ниже — крошечная
строка «Open source · Self-hosted · LLM любого OpenAI-compatible
провайдера».

**Справа:** **главный визуальный элемент** — анимированная демо
approve-flow (см. §8.1). Это hero-shot. На него тратим больше всего
визуальных усилий.

Высота секции — 100vh на десктопе.

### 5.2. Проблема — **P0**

Заголовок типа «Знакомая ситуация?»

Три карточки в ряд (или зигзаг):

1. **`kubectl exec` на свой страх и риск.** Подключение к pod, ручной
   `cat`, `tail`, `journalctl`. Каждая команда — потенциальный `rm`.
2. **Ansible playbook'и для диагностики.** Долго писать, страшно
   запускать, версионируется отдельно от ситуации.
3. **`sosreport` / `must-gather`.** Тонна данных, разбираешь сам ночью,
   половина не нужна.

Каждая карточка — иконка-мем (см. §7.3) + 1 заголовок + 2–3 строки.

### 5.3. Решение — **P0**

Одна большая визуально-выделенная карточка. Заголовок:
«Recon: расследование под вашим контролем».

Текст ~80 слов. Что говорим:
- LLM (Claude / любая OpenAI-compatible) ходит по парку через
  read-only коллекторы.
- Каждый шаг — карточка для оператора, который её аппрувит.
- На выходе — структурированный post-mortem с findings и доказательствами.
- Read-only гарантирован конструктивно, не «политикой».

### 5.4. Как это работает — **P0**

Заголовок «Один цикл расследования».

**Анимированная диаграмма** (см. §8.2) с 4 шагами:

```
[1. ВЫ ВВОДИТЕ ЦЕЛЬ]
     ↓
[2. LLM ПРЕДЛАГАЕТ ШАГ]    ← один tool_use за раз
     ↓
[3. ВЫ АППРУВИТЕ]          ← с возможностью edit/skip/end
     ↓
[4. АГЕНТ СОБИРАЕТ ДАННЫЕ] ← read-only коллектор
     ↓
[5. LLM АНАЛИЗИРУЕТ → НОВЫЙ ШАГ ИЛИ FINDING]
     ↺ цикл
```

Под диаграммой — краткое объяснение каждого шага в карусели или
списке.

### 5.5. Возможности — **P1**

Заголовок «Что внутри».

**Tab-набор или сетка карточек** (предпочтительно сетка 2×3 или 3×2):

1. **11 read-only коллекторов** — список с краткими описаниями. Каждый
   = chip с tooltip. `system_info`, `dns_resolve`, `net_listen`,
   `net_connect`, `net_ifaces`, `systemd_units`, `journal_tail`,
   `process_list`, `file_read`, `disk_usage` + ещё один. Возможность
   расширять через релиз агента.

2. **Investigation step-by-step.** Approval card, edit JSON, skip, end,
   inject hypothesis. Скриншот тёмного UI хаба.

3. **Findings с доказательствами.** Каждый finding ссылается на
   `task_id` реального наблюдения. Pin / ignore. Экспорт в Markdown.

4. **Парк хостов.** Enrollment через bootstrap token, mTLS из коробки,
   labels, статусы online/offline/degraded.

5. **Audit-лог.** Каждое действие оператора и каждое решение модели
   записывается. Кто аппрувил какой шаг и когда.

6. **Любой OpenAI-compatible LLM.** OpenRouter из коробки. Можно
   self-hosted vLLM, можно LiteLLM, можно raw OpenAI. Не привязаны к
   одному вендору.

### 5.6. Почему это безопасно — **P0 (killer section)**

Заголовок «Read-only по конструкции, не по политике».

**Выделенная секция с другим фоном** (контрастный блок).

Визуализация **5 слоёв** (см. §7.4 — иллюстрация). Под ней — текстом:

1. **Протокол.** `HubMsg` содержит только `CollectRequest` /
   `CancelRequest` / `ConfigUpdate`. В нём физически нет verb'а, который
   бы что-то менял на хосте. Хаб не может попросить агента «сделать».
2. **Каталог коллекторов.** Compiled-in список read-only функций.
   Никаких runtime-плагинов, никакого `eval`. Новый коллектор = новый
   релиз бинарника агента.
3. **Exec gateway.** Если коллектор всё-таки запускает команду — она
   проходит через whitelist разрешённых бинарей с валидированными
   аргументами.
4. **OS-уровень.** Systemd unit агента: `ProtectSystem=strict`,
   `MemoryDenyWriteExecute`, минимальный `CapabilityBoundingSet`.
   Сам процесс не имеет capabilities на запись.
5. **CI lint.** В коде collectors/ запрещены destructive syscalls и
   импорты — проверяется в CI на каждом PR.

Финальная фраза: **«ИИ не может выполнить команду, которой нет в
системе.»**

### 5.7. Технологии и развёртывание — **P1**

Заголовок «Один бинарник на хост, один на хаб».

Список с иконками:

- **Single Go binary** на каждой стороне (~28MB, без CGO, без
  зависимостей).
- **SQLite** хранилище в хабе. Никакого Postgres-кластера.
- **mTLS** между хабом и агентом обязателен. Сертификаты выдаёт хаб.
- **Self-hosted.** Ваши данные не покидают периметра — только
  сжатые рассуждения уходят в LLM-провайдер (или вообще никуда, если
  vLLM локально).
- **Systemd-юниты**, nginx config, install runbook — в комплекте.
- **Quickstart 5 минут** — линк на `deploy/docs/install.md`.

### 5.8. Сценарии использования — **P1**

Заголовок «Когда это полезно».

3–5 коротких сценариев. Каждый — карточка с иконкой и 2–3
предложениями.

- **Cron на k8s падает по ночам** — натравить Recon на namespace,
  получить причину к утру.
- **Агент Prometheus отвалился на 3 хостах** — найти, что у них общего.
- **DNS «иногда» не резолвится** — пройти стек resolv → systemd-resolved
  → нетворк.
- **Onboarding нового SRE** — пусть смотрит investigation как
  пошаговый туториал «как мы дебажим».
- **Post-mortem после инцидента** — финальный экспорт в Markdown как
  готовый раздел отчёта.

### 5.9. FAQ — **P1**

5–8 коротких вопросов-ответов. Раскрывающиеся `<details>`.

Примеры:

- *Можно ли запретить ИИ некоторые коллекторы?* — Да, через config
  whitelist на хабе.
- *Какие LLM поддерживаются?* — Любой OpenAI-compatible chat/completions
  с function calling. По умолчанию OpenRouter с `claude-sonnet-4.5`.
- *Что уходит во внешний LLM?* — Только сжатые описания результатов
  (~500–2000 токенов). Сырые артефакты остаются в хабе.
- *Это open source?* — [ответ исходя из реальности проекта; если да —
  ссылка, лицензия].
- *Сколько стоит?* — Сам Recon — [free / commercial — заполнить].
  LLM-токены — по тарифам провайдера.
- *Можно ли с air-gapped инфрой?* — Да, если поднять локальный
  vLLM/LiteLLM как backend.
- *Что насчёт Windows?* — Нет. Только Linux.
- *Как делается обновление агента?* — Через пакетный менеджер или
  замена бинарника + restart unit.

### 5.10. Footer / контакт — **P0**

Очень компактный.

```
RECON — read-only диагностика для Linux-парка под управлением LLM.

Заинтересовало? Напишите: hello@recon.ru
GitHub · Docs · Architecture

© 2026
```

Никаких «sign up for newsletter», социальных кнопок, copyright walls.

## 6. Визуальная система

### 6.1. Цветовая палитра

База **тёмная**, консистентная с хабом (см. `DESIGN_HUB.md` §6.1), но с
**ярче accent-цветами** для marketing-настроения:

```css
:root {
  /* База — как в хабе */
  --bg-0:        #0a0b0d;
  --bg-1:        #111215;
  --bg-2:        #171a1f;
  --fg-0:        #e6e8eb;
  --fg-1:        #a8adb5;
  --fg-2:        #6b7079;

  /* Marketing accents — ярче чем в продукте */
  --accent-1:    #7aa2ff;   /* основной — primary, ссылки */
  --accent-2:    #4ec9a4;   /* mint — для «безопасности» секций */
  --accent-3:    #e5b567;   /* amber — для warning/проблемы */
  --accent-4:    #c3a6e0;   /* lilac — для AI/LLM акцентов */

  /* Градиенты для иллюстраций */
  --grad-aurora:  linear-gradient(135deg, #7aa2ff 0%, #c3a6e0 50%, #4ec9a4 100%);
  --grad-warm:    linear-gradient(135deg, #e5b567 0%, #e06c75 100%);
  --grad-cool:    linear-gradient(135deg, #7aa2ff 0%, #4ec9a4 100%);
}
```

Каждая секция может «принадлежать» одному acc-цвету как теме —
например, Безопасность = mint, Проблема = amber, Как работает =
aurora gradient. Это вносит ритм и читаемость.

### 6.2. Типографика

- **Sans display (заголовки):** `Geist`, `Inter Display`, fallback
  system sans. До 56px на hero.
- **Sans body:** `Inter`, размеры 16–18px (это не density-density как
  в хабе — здесь читается с дивана).
- **Mono (код, имена, термины):** `JetBrains Mono`, `Geist Mono`.
- **Type scale:** 12 / 14 / 16 / 18 / 24 / 32 / 48 / 56.
- **Line-height:** 1.5–1.6 для body, 1.1–1.2 для display.
- **Tracking:** display чуть отрицательный (-0.02em), body 0.

### 6.3. Сетка и spacing

- Контейнер max-width 1200px по центру; full-bleed только для
  background-полос секций.
- Базовая единица 8px. Spacing: 8/16/24/32/48/64/96/128.
- Padding между секциями — 96px desktop / 64px tablet (но мобилку
  всё равно делать; см. §11).

### 6.4. Ритм страницы

Чтобы тёмный фон не задавил, чередуем:

- `--bg-0` (deep canvas) — Hero, Footer.
- `--bg-1` (panels) — Проблема, Возможности, Сценарии.
- Цветной акцент-фон с прозрачностью — Безопасность (mint glow),
  Решение (aurora glow).
- `--bg-2` (elevated) — FAQ, Tech.

Не более 4 разных фонов за весь скролл.

## 7. Иллюстрации

### 7.1. Стилистика

**Векторные SVG**, плоские, с одним-двумя bright accent-цветами на
тёмном фоне. Линии тонкие (1.5–2px), скруглённые концы, лёгкие
gradient-fill в ключевых элементах. Запрещены:
- 3D-рендеры
- isometric «корпоративные» иллюстрации (сине-фиолетовые человечки)
- стоковые «AI brain» картинки
- glassmorphism/blur эффекты

Референсы по стилю: иллюстрации Linear changelog, Vercel docs hero,
Stripe Press, Honeycomb illustrations.

### 7.2. Главные иллюстрации

1. **Hero shot** — анимированный «approve card flow» (§8.1).
2. **5 слоёв безопасности** — стек концентрических щитов / стек
   блоков, через которые пытается пройти красная стрелка «destructive
   command» и упирается. Иллюстрация-объяснение, не декорация.
3. **Hub ↔ Agents fleet** — топология: посередине хаб (с UI-иконкой),
   вокруг 6–8 агентов, между ними mTLS-link (зашифрованная линия со
   значком замка). Можно с лёгким glow.
4. **Investigation timeline** — горизонтальная схема: Goal → step →
   step → step → Finding. Каждый шаг = карточка с rationale.
5. **Иконки коллекторов** — 11 штук, каждая 32×32, узнаваемая (network
   = провод, dns = глобус с записью, journal = свиток, и т.д.).

### 7.3. Иконки секции «Проблема»

Допустимы лёгкая ирония:
- `kubectl exec` — иконка терминала с надписью `rm -r` зачёркнутой;
- ansible — тяжёлый thick playbook с песочными часами;
- sosreport — гора файлов с лупой.

Не уходить в карикатуру — лёгкий юмор, не более.

### 7.4. Диаграмма 5 слоёв

Это **flagship illustration**. На неё стоит потратить отдельно.
Концепция:

- 5 концентрических слоёв (или вертикальный стек), каждый — щит/
  барьер с подписью.
- Снаружи бьётся красная стрелка `rm -rf /` или `DELETE FROM`. На
  каждом слое стрелка теряет цвет/энергию.
- В центре — иконка хоста с зелёной галкой.
- Подписи слоёв: Protocol · Catalog · Gateway · OS · CI.
- Анимация на hover/scroll: стрелка летит по очереди в каждый слой,
  каждый «отбивает».

## 8. Интерактивные демо и motion

### 8.1. Hero — approve flow demo

**На месте hero-визуала** — компактная имитация UI хаба, где:

1. Сверху появляется текст цели: «расследовать почему упал nginx».
2. Через 1.5 сек появляется pending tool call card («tool: systemd_units,
   rationale: проверить статус nginx.service»).
3. Через 1.5 сек кнопка `Approve` подсвечивается, click-эффект, карточка
   уезжает.
4. Появляется compact result («inactive (failed) since…»).
5. Появляется новая pending card («tool: journal_tail, rationale: смотрю
   последние логи»).
6. И финальный finding: «📌 nginx OOM-killed».
7. Цикл повторяется бесконечно (с 3-сек паузой в конце).

Реализовать через CSS animations + JS-driven state (без React). Можно
Lottie если дизайнер предпочитает. Длительность одного цикла ~12 сек.
Должно работать без mouse-interaction.

### 8.2. «Как работает» — flow diagram

Диаграмма из §5.4 анимируется по scroll-into-view:
- Шаг 1 fade-in.
- Стрелка «тянется» к шагу 2.
- Шаг 2 fade-in.
- И так до 5.
- Финальная стрелка-loop возвращается к шагу 2 (показывает цикличность).

Используем `IntersectionObserver` + CSS transitions. Без библиотек.

### 8.3. 5 слоёв безопасности — interactive

При hover/click на каждый слой:
- Слой подсвечивается своим цветом.
- Снизу появляется текстовое объяснение этого слоя.
- Красная стрелка показывает, как она «застревает» именно на нём.

На мобилке — раскрывающийся аккордеон вместо hover.

### 8.4. Коллекторы — карточки

Hover на карточку коллектора → раскрывается expandable с примером
input-schema (JSON в моноширинке) и примером результата. Без JS —
только CSS `:hover` + `transition: max-height`.

### 8.5. Motion — общие правила

- Все scroll-based анимации **только один раз** (при появлении в
  viewport), не повторяются при обратном скролле — иначе раздражает.
- Hero animation — **единственная** loop-анимация. Всё остальное
  статично пока пользователь не взаимодействует.
- `prefers-reduced-motion: reduce` отключает: hero loop, scroll
  animations, hover transforms. Оставляет: только `opacity` transitions
  на 100мс.
- Никаких **parallax** на background-элементах. Никакого «mouse follow».

### 8.6. Лимиты по перформансу

- Total page weight ≤ 600 KB (включая шрифты и иллюстрации).
- Hero animation работает при 60fps на 5-летнем ноуте.
- LCP ≤ 1.5s, CLS = 0, TBT ≤ 100ms на throttled 4G.
- Lighthouse Performance ≥ 95.

## 10. Технические ограничения

- **Single static page.** Никакого Next.js, Astro, Hugo. Один
  `index.html`, один `landing.css`, один `landing.js` (если нужен).
  Можно build-step для CSS (Tailwind CLI) — но deliverable должен
  быть статикой.
- **Fonts** — через `@font-face` локально (не Google Fonts CDN — для
  privacy и offline-fallback). 2 шрифтовых семейства максимум.
- **No tracking.** Никакой Google Analytics, Yandex.Metrica, Hotjar.
  Если нужна аналитика — Plausible/Umami self-hosted, и об этом явно
  написать в footer.
- **Lang.** `<html lang="ru">`. Английская версия — потом, отдельной
  страницей или поддиректорией `/en/`.
- **A11y.** WCAG AA минимум. Все интерактивные элементы клавиатурно
  доступны. Все цветовые сообщения дублируются текстом/иконкой.
- **SEO.** `<meta>` description, OG tags, Twitter card, structured data
  (`SoftwareApplication`), `sitemap.xml`, `robots.txt`.
- **Mobile.** Несмотря на focus desktop — лендинг **ОБЯЗАН** работать
  на 360px+. CTO часто открывает ссылку на телефоне в дороге. Не
  адаптируем = не открывают.
